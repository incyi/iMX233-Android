i.MX Android R7 Standard Release Note

Release Date: Jan 20th, 2010

Release description

The i.MX Android r7 release consists of two packages:

* i.MX Android r7 Basic release
  This release includes all necessary code, document, and tools to assist users build and run Android on i.MX based hardware board from scratch. Prebuilt images are also included for a quick trial on Freescale i.MX reference development boards. This basic release includes all Freescale porting (kernel patch, HAL implementation) and enhancement (hardware acceleration, bug fix, performance tunning) based on open source Android code.
  
  Except for some proprietary modules and libraries from 3rd parties, most of the deliverables in this standard release are provided in source code.

* i.MX Android r7 codec release
  This codec release includes Freescale proprietary OpenMAX core, file parsers (for example avi) and SW audio/speech/image codec optimized for ARM11/CortexA8 (for example mp3, aac). Prebuilt images (based on standard release, with Freescale proprietary codec built in) are also included for a quick trial on Freescale i.MX reference development boards. No source code is provided in this codec release. Only libraries and header files are provided. 

This release note applies to the r7 Basic release. There is another release note for the r7 codec release. 

Note: Freescale hardware based video codecs are delivered in the Basic release as well as video display optimization. Therefore, even with the Basic release, smooth cpu-less video playback can be achieved (for example Freescale hardware based video codec + parser/codec in Android). However, the existing parser/codec in Android have many limitations and conformity/stability issues. The Freescale Android codec release may help to achieve better quality on multimedia features.

The r7 release base on v2.6.28 kernel and eclair branch of Google Android repo.

Wakeup Issues from Keyboard or from WVGA

Waking up from an action or key press from the keyboard does not always works. Waking up from WVGA does not respond.

 

 

NFS and Uboot Issues

Trying to use NFS to boot the system using uboot. The issue seemed to be a failure when sending the DHCP Discover request with an empty Parameter Request List.

Add below lines in uboot config file (myAndroid/bootable/bootloader/uboot-imx/include/configs/mx51_bbg_android.h)

#define CONFIG_CMD_DHCP

+#define CONFIG_BOOTP_SUBNETMASK

+#define CONFIG_BOOTP_GATEWAY

+#define CONFIG_BOOTP_DNS

#define CONFIG_CMD_MII

rebuild uboot

 

Screen Timeout: After a period of time, the system goes to suspend mode. The procedure to disable the screen timeout does not work.

Going to Main menu à Setting à Sound & display à Screen Timeout à Never Timout option, does not exist on R7

 

 

Codec formats (extensions) recognized by Gallery Application

After applying the information from the Codec User’s Guide, streams with extensions like AVI, or ASF are not seen by the Gallery Application; but the formats are supported.

Changing the extension from name of the stream to mp4 (only the extension in the name of the stream, it does not mean changing the stream to other format) results the stream visible for the Gallery Application and then it can be played.

Known Hardware issues

This section describes the specific notes for EVK (Babbage 3.0) board. 


i.MX51 EVK (Babbage 3.0) board

Features

Category

Description

Resolution/Workaround

General

Hardware

The GPIO PINs of some modules conflict in the EVK board.

To avoid GPIO conflict in initialization phase, more command options are provided in boot up command line “exec …” to indicate which module should be initialized:

 

“w1”:  to avoid the conflict between 1Wire and S/PDIF. If “w1” is added in the command line, 1-Wire device is initialized. Otherwise, S/PDIF will be used.

 

Boot

Hardware

Sometimes the board can not boot from SPI-NOR or MMC on EVK3.0.

Ensure Hardware fix ECO24315 is applied on your board.

Display

Configuration

The default display expected is a DVI monitor but these options are available.

The default resolution is 1024x768 (without reading the monitor preferences stored in the EDID).  If a different resolution is desired, it can be specified on the boot command line with the command: “video=mxcfb:1024x768-16M@60" where 1024x768 is the resolution desired.  If more than one display is connected, this is applied to all.  Also, the IPU limit is 4 MB which has to be divided between monitors and any video that is run, thus 1280x1024 works if there is just one monitor and no video is played.

· “wvga”: use  this command option for a WVGA panel.

· “vga”: use this command option for a VGA panel

· “mitsubishi_xga”: use this command option for a MITSUBISHI LVDS panel

· “tv” this command can be used when a TV is connected through HDMI using the DVI connector and a DVI/HDMI converter for 720p output.

 

WVGA

Hardware

WVGA backlight is powered up automatically when the power supply is applied. So before BSP initialize WVGA panel, unexpected color is observed

Currently no workaround is applied. In addition, to use WVGA panel, do as follows:

1.       Connect WVGA panel to 16bits port.

2.       Enter “Device Drivers -> Graphics Support” in kernel menuconfig. Disable “Epson VGA Panel” and enable “CLAA WVGA Panel”

3.       Add the option "wvga" in the boot-up command line.

Audio/DVI

Hardware

Some DVI monitor/cable configurations can cause the audio to not be able to load due to I2C conflict. 

Reading the EDID is broken on EVK boards due to the use of an unsuitable level shifter in the board design.  Customers are urged to not follow the reference design for circuitry to read the EDID. 

The software has disabled the DVI EDID reading feature.  If audio fails when a DVI monitor is connected, the I2C bus to the DVI connector may need to be cut.

DVI/VGA

BSP/hardware

Video playback fails if DVI resolution exceeds 1280X720.

Due to a hardware limitation, current software can not handle resolutions of 1280x720 pixels or above. The display resolution can be specified with a kernel command option such as “video=mxcfb:1024x768-16M@60".

FEC

Hardware

Sometimes the board cannot boot up if a network cable is plugged into the FEC port.

Rework the board to connect GND_ENT to GND.

Camera (OV3640)

Hardware

Camera cannot work. I2C error is reported

Need to orient the camera sensor with the lens toward the main board. The hardware rework for I2C is needed.

Camera

Hardware

ENGcm09128: Under certain conditions, the DMA controller in the IPU may cause image corruption. This applies to all IPU image sources, including both an external smart camera sensor and general images stored in memory.

The problem most frequently occurs with non-interleaved YUV420 formatted images that are rotated. The problem applies to WRITE channels only. READ channels are not affected.

Only interleaved YUV and RGB output pixel formats are supported by IPU idmac channels for i.MX51 TO2 chips. V4l2 capture and V4l2 do not support non-interleaved output pixel format. See Test Document to get the detailed camera test steps.

Audio

Hardware

ENGR00114234 ALSA: Can not capture voice from headset with microphone in J13. Possibility: 100%

J13 is connected to microphone and the right headphone channel. Ensure the headset used can support it if using it for microphone test.

Recommend to plug a tiny microphone to  the microphone connector X1 on the backside of the board.
